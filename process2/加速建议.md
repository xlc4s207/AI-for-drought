# 根系土壤湿度（SMrz）骤旱检测代码审查与加速建议

> 适用对象：你提供的 `骤旱检测主程序 - 完整修正版 v2`（Python + netCDF4 + multiprocessing）。  
> 目标：**先检查逻辑**（是否存在潜在误判/边界问题/输出一致性问题），再给出**可落地的加速思路**（从“几行改动的快赢”到“结构性重构”的方案）。

---

## 1. 程序整体逻辑概览

你的主流程可以概括为：

1. `main()` 中按纬度行（lat row）构造任务 `tasks`，使用 `multiprocessing.Pool` 并行。
2. 每个 worker 执行 `process_single_row()`：  
   - `read_row_all_years()`：逐年打开 netCDF，读取该纬度行、某经度区间的 `SMrz`，拼成 `(T_total, n_lon)`。
   - 对每个经度像元 `j`：调用 `analyze_pixel_v2(pixel_series, dates)` 检测事件。
3. 将每行结果回传给主进程：写入 `total_freq / yearly_freq`，并把每像元最多 `MAX_EVENTS_PER_PIXEL` 个事件写入 `all_events_data`。
4. 最终输出 GeoTIFF（总频率 + 年度频率）以及事件详情 NetCDF。

核心算法在 `analyze_pixel_v2()`：

- 有效性筛选（有效比例 < 0.5 的像元认为无效）。
- 计算**后向**滑动平均 `sm_ma`（只用当天及之前）。
- 用 `sm_ma` 按 DOY（day-of-year）统计长期分布，求 P20/P40（或 config 中的 low/high）。
- 用状态机检测：  
  - `NORMAL`：记录最后一次 `>P40` 的日子为 `wet_start_idx`；当首次 `<P20` 时判断 onset 天数是否在 5–30 天；满足则进入 `DROUGHT`。
  - `DROUGHT`：持续 `<P20`，回升到 `>=P20` 则事件结束；干旱期内连续 3 天 NaN 则中断事件。
- 事件输出字段：起止年/DOY、onset_days、drought_days、onset_drop、onset_rate、intensity。

## 2. 逻辑审查：可能的问题与改进点

下面列出我认为“值得你确认”的点（有些不一定是 bug，但可能导致误判或输出不一致）。

### 2.1 `event_count` 对“无效像元”的编码不一致（**建议修正**）

你在 `main()` 初始化：

- `total_freq` / `yearly_freq`：用 `NaN` 表示无效像元（✅一致）。
- 但 `all_events_data['event_count']` 初始化为 **0**：
  ```python
  'event_count': np.zeros((n_lat, n_lon), dtype=np.int16)
  ```

这会导致：**无效像元在 NetCDF 里 event_count=0**，与“有效但没有事件”的像元无法区分（因为它也会是 0）。  
而你创建 NetCDF variable 时 `fill_value=-1`，但你写入的数组本身是 0，所以 fill_value 不会生效。

**建议：**
- 初始化为 -1：
  ```python
  all_events_data['event_count'] = np.full((n_lat, n_lon), -1, dtype=np.int16)
  ```
- 对于“有效像元但 0 个事件”，显式写 0；对于无效像元保持 -1。

这是一个非常实用的修正点，因为它直接影响后处理统计。

### 2.4 `doy not in percentiles` 时直接跳过（可能让状态机“穿越”缺阈值日）

在循环里：
```python
if doy not in percentiles:
    continue
```

这会导致某些 DOY（比如全是 NaN 或样本太少导致未建立 percentiles）被完全跳过：
- `NORMAL`：不会更新 wet_start_idx 或触发干旱。
- `DROUGHT`：不会结束干旱，也不会累积强度。

如果这种“无阈值日”连续出现，状态机可能产生不符合预期的持续期。

**建议：**
- 若 `doy` 缺失阈值，可以把它视为缺测，按 NaN 的规则处理（例如干旱期内累计 NaN 并可能中断）。

### 2.5 DOY=366 的样本量很小（阈值不稳定风险）

对于非闰年没有 DOY=366；闰年可能只有少数年份。这样 P20/P40 的估计可能非常不稳定，导致事件在闰年末尾附近有跳变。

**可选处理：**
- 对 DOY=366 用 DOY=365 的阈值替代；或
- 用滑动 DOY 窗口（例如 ±7 天）来估计“日历日百分位”（更平滑、更稳健）。

## 3. 为什么慢：主要性能瓶颈定位

从代码结构上看，慢的来源通常是“CPU 端 Python 循环 + I/O 频繁打开文件”的叠加。

### 3.1 I/O 开销：每行×每年反复打开 netCDF 文件

`read_row_all_years()` 对每个 row 都会对 YEARS 循环：
- `with nc.Dataset(...) as ds` 打开文件
- 读取该 row 的全年时间序列

若有 `n_lat` 行、`n_years` 年，则总打开次数约 `n_lat * n_years`。在机械硬盘、网络盘或并发较高时非常耗时。

### 3.2 计算开销：逐像元的滑动平均是 O(T*window)

`calculate_backward_moving_average()` 对每个像元 `for i in range(n)`，每次都切片并 `np.nanmean`：
- 时间复杂度约 `O(T * window)`  
- 这段会成为全局热点（尤其是像元数量大时）。

### 3.3 计算开销：每像元构造 `doy_values` dict + 逐 DOY percentile

对每个像元都做：
- 遍历全时序构造 `doy_values[doy].append(...)`
- 再对 1..366 逐 DOY 计算 percentile

dict/list 的 Python 解释器开销很明显。

### 3.4 计算开销：intensity 每个事件又嵌套遍历

每个事件：`for j in range(drought_start_idx, event_end + 1)` 再叠加 DOY 查表与 if。  
如果一个像元事件较多或持续期长，这部分也会变成热点。

## 4. 加速方案（从易到难）

我按“改动成本”把方案分层，建议你从上往下逐步实施，并用 test-mode 对比结果一致性。

### 4.1 低成本快赢（几行改动级别）

#### A) dates 不要在每个 row 重建
`read_row_all_years()` 每次都构造 `all_dates`（year,doy）列表，这在每行都会重复。  
可以在 `main()` 里根据“每年天数”一次性生成 `dates` / `year_arr` / `doy_arr`，然后作为全局常量供 worker 使用。

#### B) 给 `imap_unordered` 设置 chunksize
默认 chunksize 可能太小导致调度开销高：
```python
for result in pool.imap_unordered(process_single_row, tasks, chunksize=4):
    ...
```
chunksize 取 2–16 通常就能减少开销（具体看 n_lat 和机器核数）。

#### C) netCDF4 关闭自动 mask/scale（减少 masked array 开销）
目前你在读取后又检查 `hasattr(row_data, 'mask')` 并 filled。  
你可以尝试：
- 在打开 ds 后 `ds.set_auto_mask(False)`（或对变量 `var.set_auto_mask(False)`）  
- 并手动用 `_FillValue` 替换为 NaN

这能减少 masked array 的构造成本（实际收益取决于数据缺测比例）。

### 4.2 中等成本（函数替换级别）：把滑动平均从 O(T*window) 改为 O(T)

下面给一个**等价思路**：用 cumulative sum 计算后向窗口均值（忽略 NaN），同时计算窗口内有效值计数。

> 注意：你原始逻辑要求“有效值数 >= window//2 才给均值”。  
> 下面的实现也保留这一规则（对靠近序列起点的 i，窗口长度小于 window 时，依然要求 >= window//2 ——与原逻辑一致）。

```python
def backward_moving_average_fast(x, window, min_valid=None):
    import numpy as np
    if min_valid is None:
        min_valid = window // 2

    x = x.astype(np.float64, copy=False)
    valid = ~np.isnan(x)
    x0 = np.where(valid, x, 0.0)

    cs = np.cumsum(x0)
    cc = np.cumsum(valid.astype(np.int32))

    # rolling sum/count for backward window
    cs_shift = np.concatenate(([0.0], cs[:-window])) if window < len(x) else np.zeros_like(cs)
    cc_shift = np.concatenate(([0], cc[:-window])) if window < len(x) else np.zeros_like(cc)

    win_sum = cs - cs_shift
    win_cnt = cc - cc_shift

    ma = np.full_like(x, np.nan, dtype=np.float64)
    ok = win_cnt >= min_valid
    ma[ok] = win_sum[ok] / win_cnt[ok]
    return ma
```

这通常是你全程序里“最明显的单点加速”。

进一步，你还能对整个 `(T, n_lon)` 的 row 矩阵一次性做 rolling mean（沿 axis=0），把 moving average 这一块变成纯 numpy 向量化。

### 4.3 中高成本（结构优化）：按“行（row）”批量算百分位，而不是每像元 dict

你现在对每个像元单独做 DOY 分组与 percentile。  
如果每年时间维度一致（或你可以把每年补齐到 366 并用 NaN 填充），那么可以把 row 的 `sm_ma` reshape 为：

- `sm_ma_yearly`：`(n_years, 366, n_lon)`  
然后直接：
```python
p20 = np.nanpercentile(sm_ma_yearly, 20, axis=0)  # (366, n_lon)
p40 = np.nanpercentile(sm_ma_yearly, 40, axis=0)
```

这样做的好处：
- **避免了每像元的 dict/list 构造**
- percentile 计算完全在 numpy 内部完成
- 可以一次得到整行所有像元的阈值

### 4.4 中高成本：把 intensity 变成 O(1) 区间查询（用前缀和）

如果你已经有了每个时间点对应的 P20（按 DOY 映射），可以构造：
- `deficit = max(P20(t) - sm_ma(t), 0)`（NaN 视为 0）
- `cum_deficit = cumsum(deficit)`

那么某个事件 `[start, end]` 的 intensity 就是：
```python
intensity = cum_deficit[end] - cum_deficit[start-1]
```
无需在事件内部逐日循环。

这对“事件持续期长/事件多”的场景会很有用。

### 4.5 高成本但潜在最高收益：把状态机也按行向量化（一次处理 1000+ 像元）

目前你是：row 内对每个像元做一遍完整的时序循环。  
更高效的方式是：

- 外层只对时间 `i=0..T-1` 循环一次
- 用 `state[n_lon] / wet_idx[n_lon] / drought_idx[n_lon] / nan_cnt[n_lon]` 这些数组维护每列像元的状态
- 在每个时间步用布尔掩膜更新状态（numpy 广播）

这样复杂度从：
- `O(n_lon * T)`（你现在是这个，而且还有很多 Python 分支）
变为：
- `O(T)` 次“矢量化更新”，每步处理 n_lon 个像元（主要成本在 numpy 层）

缺点是实现更复杂（尤其是事件列表的写入），但如果你的网格很大，这是最可能获得数量级提速的方案。

### 4.6 另一个方向：使用 Numba/JIT 编译像元级循环

如果你更希望保留像元级状态机（写起来直观），可以考虑：
- 把 `analyze_pixel_v2()` 改写为只用 numpy 数组、避免 dict/list（或把 percentile 预先算好并以数组传入）
- 然后用 `numba.njit` 编译

这对 Python for 循环和分支非常有效，通常能把 CPU 热点降到可接受范围。

## 5. 建议的实施顺序（最稳妥）

1. **修正 `event_count` 无效像元编码**（第 2.1 节）。
2. 给 `imap_unordered` 加 `chunksize`，并把 dates 预生成（4.1）。
3. 把 moving average 替换为 O(T) 版本（4.2）。
4. 如果还慢：把 percentile 计算改为“按行批量 nanpercentile”（4.3）。
5. 再慢：用前缀和加速 intensity（4.4）。
6. 最后才考虑：行级向量化状态机 或 numba（4.5/4.6）。
